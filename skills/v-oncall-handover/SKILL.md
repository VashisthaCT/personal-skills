## Step 1 — Compute the week window

If `--week-start` not given: compute previous Monday in IST (today if today is Monday, else most recent past Monday).

```
start_ist = <weekstart>T00:00:00+05:30
end_ist   = <weekstart>+7dT00:00:00+05:30
start_utc = start_ist - 5h30m

## Step 3 — Verify each PD (false-alert filter)

For each PD bot post in scope, run all of these checks:

1. **Read its Slack thread.** If any human reply contains the verbatim text "false alert" or "test alert" → bucket as FALSE / TEST.
2. **Auto-resolve pattern.** If TTR < 5 min, no human ack, no human reply → likely FALSE flap. Confirm via #4.
3. **Pull alert payload** via `mcp__clarity-pagerduty__get_incident_alerts`:
   - If `cube.kind = anomaly` and query is `<> 4` or `< 4` on call-rate → traffic-dip detector. If `value < anomaly_prediction` → FALSE (just a quiet period). If during a known outage window → REAL (traffic genuinely dropped).
   - If alert annotation contains `Test alert` or `Notification test` → TEST.
   - If alert is `cube.kind = static` (e.g. `Error percentage (High)` rule_id 1009, `> 10%` or `> 25%`) → REAL static-threshold breach.
4. **Same-pattern repeats:** group consecutive alerts with the same title pattern → one row with `Count = N`, time range as `start – end`.

Bucket into:
- **Real** — goes in the per-region "Real" table.
- **False alerts** — goes in the **single combined "Noise/False Alerts" table** (covers both IND and KSA — many anomaly fires are KSA-named but post to the IND PD service after the receiver-group re-route). Include a `Region` column. Add a verification quote.
- **Test** — drop entirely (planned, not noise to surface).

**KSA RPM:**
```

```

**KSA latency (mean, ms):**
```
sum by (span_name) (rate(cube_apm_latency_sum{service=~"einvoicingbe-prod-oci.*", span_kind="server", span_name=~"<same regex>"}[15m])) / sum by (span_name) (rate(cube_apm_latency_count{service=~"einvoicingbe-prod-oci.*", span_kind="server", span_name=~"<same regex>"}[15m]))
```

**IND RPM:**
```
sum by (span_name) (rate(cube_apm_calls_total{service=~"einvoicingbe-prod-(http|bulk-http)|einvoicing-integrations-prod-http", span_kind="server", span_name=~"WebTransaction/SpringController/v[12]/(einvoices|eInvoice|ewaybill|govt).*|WebTransaction/SpringController/(ping|OperationHandler).*|WebTransaction/Uri/metrics.*"}[15m]) * 60)

**IND latency:** same service+span_name regex as IND RPM, same `_sum/_count` pattern as KSA latency.

### Width profile auto-detection

### Adjusting column widths (one-time per table type)

If a column on a newly-pushed table is too narrow:
1. Drag the column borders in Coda UI.
2. Trigger an HTML export of the page (`POST /apis/v1/docs/{doc}/pages/{page}/export` with body `{"outputFormat":"html"}`, then poll + GET the downloadLink).
3. Grep the export for `<th style="width:Npx"...data-coda-column-id="...">ColName</th>` for that table's grid-id.
4. Update `data/coda_table_widths.yaml` under the appropriate key.
5. Future pushes auto-use the new widths.

Token: generate at <https://coda.io/account> (API Settings → Generate API token, Full access). Save to `~/.coda_token` (chmod 600) once.

### Don't (Coda-specific)
- Don't push as `format: "markdown"` — column widths get squished, especially on tables with one long-prose cell beside a short-numeric cell.
- Don't try `<col style="width:..">` or `<colgroup>` — Coda strips both.
- Don't use the public API's `/duplicate`, `/copy`, `/clone` endpoints for tables — they don't exist (404).
- Don't try to upload chart PNGs — Coda has no public file-upload-to-canvas endpoint. The numeric metric tables convey the same data; charts stay local.
